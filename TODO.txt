 Critical Bugs
                                                                                                                                                                                  
  1. hardDropPiece leaves the piece in .locking state forever if the lock timer is cancelled

  GameController.swift:283-287 — hardDropPiece sets state = .locking but doesn't call makeLockTimer(). If the drop timer fires while the lock timer hasn't been started yet, the state
  machine can get confused. More importantly, if the lock timer somehow gets cancelled (e.g. via pause during the brief window), the piece never locks:                           

  public func hardDropPiece() {
      guard isPlaying else { return }
      while canMoveDownPrivate() { currentY += 1 }
      state = .locking  // ← no makeLockTimer() call!
  }

  Fix: Call makeLockTimer() after hard drop, or inline the lock logic.

  Bugs / Correctness Issues
                                                                                                                                                                                  
  4. ConsoleGameUI is @unchecked Sendable but isn't truly safe                                                                                                                    

  ConsoleGameUI.swift:7 — The input property and logger are captured by @Sendable closures without synchronization. ConsoleRenderer is also @unchecked Sendable                   
  (ConsoleRenderer.swift:10). These work in practice because the closures run on the same actor context, but the compiler can't verify this.


  5. Input handler doesn't support arrow keys or escape sequences                                                                                                                 

  ConsoleInputHandler.swift:51-64 — Only single characters are handled. Terminal arrow keys send escape sequences like \[A (4 bytes). Pressing Up gives � (byte 27 = \u{1b} = ESC,
  triggering pause instead of the intended action). Consider adding escape sequence parsing.

  Performance Issues
                                                                                                                                                                                  
  7. pieceBlocks linear search per cell

  ConsoleRenderer.swift:51 — For every cell in the 200-cell grid, .first(where:) iterates pieceBlocks (4 items). Tiny now but O(n*m) per render. Use a Set or 2D lookup:          

  let pieceSet = Set(data.pieceBlocks.map { ($0.x, $0.y) })
  // then: if pieceSet.contains((x, y)) { color = ... }

  8. getTerminalSize() called every render

  ConsoleRenderer.swift:18 — Terminal size rarely changes. Cache it with lazy evaluation.

  UX / Gameplay Issues
                                                                                                                                                                                  
  10. Hard drop doesn't give score

  @GameController.swift: The piece lands in .locking state and the lock timer eventually calls lockPiecePrivate() and clearLinesPrivate(). So lines are cleared and scored, but there's a noticeable delay after a hard drop before anything happens. The user sees the piece "hover" in locking state without immediate feedback. Consider having hardDropPiece inline the lock+clear logic.                                                                                                                                                          


  11. No backspace / unpause on hard drop while paused                                                                                                                            

  GameController.swift:226-234 — Hard drop during .paused does nothing (continue). Traditional Tetris allows unpause with a dedicated key (often Enter or P), not hard drop.      


  12. storePlayerName accepts empty strings

  ScoreStorage.swift:101-118 — No validation on input. "User: " with nothing after it is a valid outcome.

  Code Quality / Refactoring Suggestions
                                                                                                                                                                                  
  13. spawnNextPiece creates [TetrominoShape] array on every call

  GameController.swift:339 — A static constant allShapes would avoid allocation. Same pattern in init() and resetGame().                                                          


  14. GameState lacks Sendable conformance                                                                                                                                        

  GameState.swift:1 — Passed through @Sendable closures as GameSessionState.state. Add : Sendable (and consider RawRepresentable).                                                


  15. OverlayLine nested inside ConsoleRenderer

  ConsoleRenderer.swift:149 — Nested structs aren't reusable. Extract to its own file or at least make it public.


  16. ConsoleRenderer.render() mixes layout calculation with output generation

  ConsoleRenderer.swift:17-141 — The 140-line render function handles: positioning, borders, grid, score, status, next-piece preview, controls, and game-over overlay. Split into
  dedicated methods (renderGrid(), renderOverlay(), renderControls(), etc.).


  17. Game over flow exits the entire app instead of offering restart

  ConsoleGameUI.swift:42-44 — onGameFinished signals a semaphore that terminates the entire app. Traditional Tetris loops: game over → new game. The GameController.restart()
  method exists but the UI never calls it — after game over, the user must relaunch the binary.
